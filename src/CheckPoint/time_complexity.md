# ⏱️ 시간초과 방지 사고법 (Algorithm Thinking Guide)

## 🔥 핵심 원리
> 시간초과의 대부분은 **같은 계산을 반복해서 수행**하기 때문에 발생한다.

👉 목표:
- **중복 계산 제거**
- **필요한 정보만 유지**
- **각 원소를 최소 횟수만 처리**

---

# ❌ 시간초과 나는 코드 특징

- 이중 / 삼중 반복문
- 같은 구간 반복 탐색
- 같은 상태 반복 호출 (재귀, BFS 등)
- 매번 최댓값 / 최솟값 재탐색
- ArrayList 중간 삭제 남용
- 불필요한 상태 저장

---

# ✅ 시간초과를 막는 핵심 사고

## 1. 같은 계산을 반복하고 있지 않은가?
👉 해결:
- 누적합 (Prefix Sum)
- DP / 메모이제이션
- visited 배열

---

## 2. 전체를 다시 보지 말고 변화만 처리하자
👉 해결:
- 투 포인터
- 슬라이딩 윈도우
- 우선순위큐
- sweep line

---

## 3. 값 기준 반복 → 원소 기준 1회 처리
👉 해결:
- 단조 스택
- 이전/다음 작은 값
- 각 원소 1번 처리

---

## 4. 정렬로 탐색 범위를 줄일 수 있는가?
👉 해결:
- 그리디 + 정렬
- 투 포인터
- 크루스칼

---

## 5. 최적 후보를 빠르게 찾을 수 있는가?
👉 해결:
- PriorityQueue
- TreeSet / TreeMap
- Deque

---

## 6. 상태를 줄일 수 있는가?
👉 해결:
- 상태 압축
- prev 배열
- 불필요한 정보 제거

---

## 7. 자료형 / 입출력도 중요하다
👉 체크:
- BufferedReader 사용
- StringBuilder 사용
- int vs long 확인
- remove 비용 확인

---

## 8. 복잡도를 숫자로 계산하라

| 입력 크기 | 허용 복잡도 |
|----------|------------|
| N ≤ 1000 | O(N^2) 가능 |
| N ≤ 100000 | O(N log N) 필요 |
| N ≤ 1e6 | O(N) 필요 |

👉 기준: 약 1억 연산

---

# 🚀 필수 패턴 매칭

| 상황 | 해결 |
|------|------|
| 구간합 반복 | 누적합 |
| 부분문제 반복 | DP |
| 정렬된 조건 탐색 | 투 포인터 |
| 최댓값 선택 반복 | 우선순위큐 |
| 이전/다음 큰 값 | 스택 |
| 연결 여부 | Union-Find |
| 최단거리 | BFS / 다익스트라 |

---

# 🧠 실전 체크리스트

1. 같은 값 여러 번 계산하고 있나?
2. 배열을 반복해서 다시 보고 있나?
3. 정렬하면 쉬워지나?
4. 포인터를 한 방향으로만 움직일 수 있나?
5. 후보를 자료구조로 관리할 수 있나?
6. 구간 정보를 누적할 수 있나?
7. 상태를 줄일 수 있나?
8. 원소를 한 번만 처리할 수 있나?
9. 최악 반복 횟수는?
10. 입력 크기에 맞는 복잡도인가?

---

# 🧩 핵심 요약

> 전체를 반복하지 말고  
> 필요한 정보만 유지하면서  
> 각 원소를 최소 횟수로 처리하자
