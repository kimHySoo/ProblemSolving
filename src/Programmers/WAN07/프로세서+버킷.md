# 프린터 문제 - 우선순위 버킷(Bucket) 방식 풀이 정리

## 문제 요약

```
1. 실행 대기 큐(Queue)에서 대기중인 프로세스 하나를 꺼낸다.
2. 큐에 대기중인 프로세스 중 우선순위가 더 높은 프로세스가 있다면
   방금 꺼낸 프로세스를 다시 큐에 넣는다.
3. 만약 그런 프로세스가 없다면 방금 꺼낸 프로세스를 실행한다.
   3.1 한 번 실행한 프로세스는 다시 큐에 넣지 않고 그대로 종료된다.
```

`location` 위치에 있던 프로세스가 몇 번째로 실행되는지를 구하는 문제.

---

## 접근 방식: 우선순위 버킷

### 핵심 아이디어

우선순위 값의 범위가 **1~9로 작고 고정**되어 있다는 점을 이용해서, 큐 전체를 매번 스캔하며 비교하는 대신 **우선순위별로 미리 분류(버킷팅)** 해두고 높은 순서대로 처리한다.

```java
LinkedList[] ll = new LinkedList[10];  // ll[1] ~ ll[9] : 우선순위별 버킷
```

각 버킷에는 해당 우선순위를 가진 프로세스들의 **원본 배열 인덱스**를 순서대로(오름차순) 저장한다.

### 왜 빠른가

- **일반적인 큐 시뮬레이션**: 하나 꺼낼 때마다 "더 높은 우선순위가 있는가?"를 확인하려고 남은 큐 전체를 스캔 → O(n²)
- **버킷 방식**: 우선순위 9 → 1 순서로 버킷 자체를 순회. 특정 버킷을 만났다는 것 자체가 "이보다 높은 우선순위는 이미 다 처리됨"을 의미하므로, 비교 없이 **버킷의 개수(`cnt`)를 통째로 더하기**만 하면 됨 → O(n)

### 처리 흐름

```java
for(int i=9; i>0; i--) {
    if(ll[i].cnt==0) continue;
    if(i != priorities[location]) {
        answer += ll[i].cnt;         // location보다 우선순위 높은 버킷: 통째로 실행되고 지나감
        idx = ll[i].findlast(idx);   // 이 버킷을 지나면서 큐의 "현재 위치" 갱신
    } else {
        answer += ll[i].find(idx, location);  // location이 속한 버킷: 정밀 계산
        break;
    }
}
```

- `idx` : 지금까지의 처리를 통해 큐가 "어디까지 돌았는지"를 나타내는 기준점
- `findlast(idx)` : 현재 버킷을 다 처리하고 나면 큐의 기준점이 어디로 이동하는지 계산
- `find(idx, loc)` : location이 속한 버킷 안에서, `loc`이 몇 번째로 실행되는지 원형 순서(circular order)로 계산

## 겪었던 오류와 원인 정리

### 1. `cnt++`의 위치 문제 (add 함수)

```java
// 잘못된 코드
void add(int val) {
    Node cur = new Node(val);
    cnt++;                    // 증가를 먼저 해버림
    if(cnt==0) { ... }        // 항상 false
}
```

**원인**: `cnt`를 먼저 증가시킨 뒤 `cnt==0`을 검사해서, 첫 add 호출에도 조건이 항상 거짓이 됨. `tail`이 아직 `null`인 상태에서 `tail.next`에 접근해 `NullPointerException` 발생.

**해결**: `cnt++`를 조건 검사 이후로 이동.

---

### 2. `find()`에서 순회 코드 누락

```java
// 잘못된 코드
for(int i=0;i<cnt;i++) {
    if(cur.val==loc) loc_idx = i;
    if(!find_last && cur.val>idx) {
        find_last = true;
        last_idx = i;
    }
    // cur = cur.next; 가 없음!
}
```

**원인**: 루프를 돌면서도 `cur`가 다음 노드로 이동하지 않아, 사실상 첫 번째 노드 값만 반복해서 검사하는 셈이 됨. `loc_idx`, `last_idx` 계산이 전부 무의미해짐.

**해결**: 루프 마지막에 `cur = cur.next;` 추가.

---

### 3. `findlast()`의 잘못된 반환값

```java
// 잘못된 코드
int findlast(int idx) {
    Node cur = head;
    if(cur.next.val>=idx) return cur.next.val;  // 최솟값을 리턴해버림
    ...
}
```

**원인**: 버킷의 모든 원소가 `idx` 이상이라면, wraparound 없이 순서대로 전부 방문되므로 **마지막으로 방문되는 건 최댓값(tail)** 이어야 함. 그런데 코드는 조건을 만족하면 **첫 번째(최솟값) 노드를 그대로 리턴**해버려서 의미가 반대로 뒤집힘.

**해결**: "idx보다 작은 원소들 중 최댓값"을 찾고, 그런 원소가 없으면 `tail.val`을 기본값으로 리턴하도록 재작성.

```java
int findlast(int idx) {
    Node cur = head.next;
    int result = tail.val;      // 기본값: 전부 idx 이상이면 최댓값
    while (cur != null) {
        if (cur.val < idx) result = cur.val;
        cur = cur.next;
    }
    return result;
}
```

---

### 4. `idx` 초기값이 0으로 설정된 문제 (가장 찾기 어려웠던 버그)

```java
int idx = 0;  // 문제의 원인
```

**원인**: `idx`는 "이미 처리된 기준 위치"를 의미하는데, 배열 인덱스는 0부터 시작하기 때문에 `idx=0`으로 두면 **"인덱스 0번이 이미 처리됐다"**는 뜻이 되어버림. 그 결과 정작 아무것도 처리되지 않은 최초 상태에서도, 인덱스 0번 원소가 `val > idx` 비교(`0 > 0 = false`)에서 "나중에 처리되는(wraparound) 그룹"으로 잘못 분류됨.

**예시**: `priorities = {9, 9}, location = 0`
- 실제 정답은 1 (큐에 (0,9),(1,9)만 있으므로 첫 번째가 바로 실행)
- 하지만 `idx=0`일 때 `find(0, 0)`을 계산하면 인덱스 0이 "먼저 그룹"이 아닌 "나중 그룹"으로 잘못 분류되어 answer=2가 나옴

**해결**: `idx` 초기값을 `-1`로 변경. 그러면 인덱스 0번도 `0 > -1 = true`가 되어 정상적으로 "아직 처리되지 않은 맨 앞 그룹"으로 분류됨.

```java
int idx = -1;  // 수정
```

## 배운 점

1. **연산자와 실행 순서**: `cnt++`처럼 사소해 보이는 위치 하나가 조건문 전체의 의미를 뒤집을 수 있다.
2. **순회 로직 검증**: 반복문 안에서 포인터(`cur`)를 실제로 이동시키는 코드가 빠지면, 로직은 그럴듯해 보여도 결과는 완전히 틀어진다.
3. **경계값(edge case)의 의미를 명확히 하기**: "이미 처리된 위치"라는 개념을 `idx`로 표현할 때, 그 값의 시작점이 0인지 -1인지에 따라 첫 번째 원소의 분류가 완전히 달라진다. 배열 인덱스가 0부터 시작한다고 해서 "처리 안 됨" 상태의 기준값도 0이어야 하는 건 아니다.
4. **직관적인 방향(최댓값 vs 최솟값) 재확인**: `findlast`처럼 "마지막으로 방문되는 값"을 구할 때, 조건을 만족하는 첫 원소가 아니라 반대편 끝(tail)을 봐야 하는 경우가 있다. 변수 이름과 실제 반환 로직이 일치하는지 항상 재확인이 필요하다.

## 참고: 시간복잡도 비교

| 방식 | 시간복잡도 | 특징 |
|---|---|---|
| 큐 시뮬레이션 (전체 스캔) | O(n²) worst case | 구현이 단순하고 버그 적음 |
| 우선순위 버킷 (본 풀이) | O(n) | 구현 복잡, 경계값 버그 위험 높음 |

이 문제는 입력 크기가 작아(보통 100 이하) 두 방식 모두 시간 내에 통과 가능하지만, 버킷 방식은 O(n) 최적화를 위해 경계값 처리에 훨씬 세심한 주의가 필요하다는 것을 몸소 확인한 문제였다.
