# 길 찾기 게임 - 오답노트

## 문제 요약
좌표가 주어진 노드들을 아래 규칙에 맞는 이진트리로 구성한 뒤, 전위순회/후위순회 결과를 반환하는 문제.

- 같은 레벨(level)의 노드는 같은 y값을 가진다.
- 자식 노드의 y값은 항상 부모보다 작다.
- 왼쪽 서브트리의 모든 노드는 x값이 V보다 작고, 오른쪽 서브트리는 V보다 크다.

**핵심 통찰**: 위 규칙은 사실상 **BST(이진 탐색 트리)의 정의와 완전히 동일**하다. y값 기준 내림차순으로 정렬해서 순서대로 삽입하면, 표준 BST 삽입 로직만으로 트리가 자동 완성된다.

---

## BST(이진 탐색 트리) 삽입 개념 정리

### 정의
왼쪽 서브트리는 모두 작은 값, 오른쪽 서브트리는 모두 큰 값이 되도록 유지하는 이진트리.

### 삽입 로직
```java
Node cur = root;
while (true) {
    if (newNode.x < cur.x) {
        if (cur.left == null) { cur.left = newNode; break; }
        cur = cur.left;
    } else {
        if (cur.right == null) { cur.right = newNode; break; }
        cur = cur.right;
    }
}
```

### 왜 이것만으로 충분한가
- **"루트부터 시작해서 한 칸씩 내려가며, 매번 새로 비교한다"**는 것이 핵심.
- `cur`이 한 칸 내려갈 때마다 이전 조상과의 좌/우 관계는 이미 "그 경로를 타고 내려왔다"는 사실 자체에 녹아있어 별도로 기억하거나 계산할 필요가 없음.
- 예: "왼쪽 서브트리의 오른쪽 자식"이든 "오른쪽 서브트리의 왼쪽 자식"이든, `cur`이 이동한 지점에서 **다시 처음부터 비교**하기 때문에 항상 정확한 직속 부모와 좌/우가 결정됨.
- 즉 "어느 서브트리에 속하는지 미리 구별"하는 로직이 필요 없다. 비교를 반복하는 것 자체가 그 구별 역할을 대신한다.

### 힙(heap)과의 차이 (헷갈렸던 부분)
| | 힙 | BST 삽입 |
|---|---|---|
| 삽입 위치 결정 방식 | 구조상 정해진 다음 빈 자리(완전 이진트리 유지)에 먼저 넣음 | **값 비교로 매번 좌/우를 선택**하며 내려가서 위치 자체를 찾음 |
| 삽입 후 처리 | 부모와 비교해 위로 올라가며 swap (sift-up) | 없음. 빈 자리를 찾으면 그걸로 끝 |
| 트리 모양 | 항상 완전 이진트리로 고정 | 값 분포에 따라 편향될 수 있음 |

BST 삽입은 오히려 **이진 탐색(binary search)**과 원리가 같다: "찾는 값이 중간값보다 작으면 왼쪽, 크면 오른쪽" 을 반복하다가 값이 없으면(null) 그 자리에 삽입하는 것.

## 시행착오 1: "레벨 단위 + 범위(range) 추적" 방식의 문제점

### 접근 방식
같은 y값(레벨)의 노드들을 한 번에 묶어 처리하고, 부모 레벨의 각 노드가 커버하는 x 범위(`left_min`, `right_max`)를 직접 계산해서 저장한 뒤, 투 포인터(`par_idx`)로 자식이 어느 부모 범위에 속하는지 매칭.

### 문제점
- `parent.x < e.x <= parent.right_max`인 경우(= par_idx를 옮길 필요 없이 바로 현재 부모의 오른쪽 자식이 되는 경우)의 분기가 누락되어 있었음. 어떤 조건에도 걸리지 않고 그냥 넘어가서 `parent`가 null로 남는 노드가 발생.
- "부모가 커버하는 x구간"이라는 정보를 직접 계산해서 들고 다녀야 하므로 로직이 복잡해지고 엣지케이스(레벨을 여러 단계 건너뛰는 경우 등)를 놓치기 쉬움.

### 교훈
BST 삽입에서는 "부모의 관할 구역"이라는 개념을 트리 구조 자체(조상 노드들의 x값)가 암묵적으로 표현하기 때문에, 별도로 range를 계산·저장할 필요가 없다. 굳이 직접 계산하려다 보니 버그가 발생한 것.

## 시행착오 2: 구현 단계에서의 실수

### 버그 A — 순회 함수 호출 누락
```java
pre = new ArrayList<>();
post = new ArrayList<>();

for(int i=0;i<n;i++) {
    answer[i][0] = pre.get(i);   // pre가 비어있는 상태에서 get(i) 호출 → IndexOutOfBoundsException
    answer[i][1] = post.get(i);
}
```
트리는 다 만들었지만 `preorder(root)`, `postorder(root)`를 실제로 호출하는 코드가 빠짐. 리스트 생성만으로는 순회가 자동으로 일어나지 않는다.

**수정**:
```java
Node root = g.get(0);
preorder(root);
postorder(root);
```

### 버그 B — 반환 배열의 shape 오류
문제가 요구하는 반환 형식은 `[전위순회 배열, 후위순회 배열]`, 즉 `int[2][n]`.

```java
// 잘못된 형태: [n][2] → [[7,9],[4,6],...] (노드별 쌍)
int[][] answer = new int[n][2];
answer[i][0] = pre.get(i);
answer[i][1] = post.get(i);
```

```java
// 올바른 형태: [2][n] → [[7,4,6,...],[9,6,5,...]]
int[][] answer = new int[2][n];
for(int i=0;i<n;i++) {
    answer[0][i] = pre.get(i);
    answer[1][i] = post.get(i);
}
```

**교훈**: 배열 차원의 순서(`[n][2]` vs `[2][n]`)를 문제 예시와 반드시 대조할 것. 컴파일과 실행은 되지만 결과 형태가 다르면 오답 처리됨.

## 제한사항 관련 주의점 (N ≤ 10,000, 깊이 ≤ 1,000)

- **삽입**: while 반복문으로 구현하면 재귀가 아니므로 안전.
- **순회**: 재귀로 구현 시 트리가 편향되면 깊이 1,000까지 재귀가 쌓일 수 있어 StackOverflowError 위험. 안전하게 하려면 스택을 직접 쓰는 반복문(iterative) 방식 고려.
  - 전위순회: 스택에 push/pop으로 간단히 구현 가능 (오른쪽 자식 먼저 push, 왼쪽 자식 나중에 push)
  - 후위순회: "역순 전위순회(root→right→left)를 구한 뒤 뒤집기" 트릭 사용
- 단, 이 문제는 트리 깊이가 최대 1,000으로 제한되어 있어 실제로는 재귀로도 대부분 통과 가능.

## 최종 정리 체크리스트

- [ ] y 내림차순(동률 시 x 오름차순 등 일관된 기준) 정렬
- [ ] 정렬된 순서대로 BST 삽입 (while 반복문 권장, 재귀 삽입 지양)
- [ ] 트리 완성 후 반드시 preorder/postorder 함수를 **호출**
- [ ] 반환 배열의 shape이 문제 요구사항과 일치하는지 확인 (`[2][n]`)
- [ ] 노드 수가 많을 경우 재귀 깊이 문제 고려 (iterative 순회 대안 숙지)
