# [오답노트] 워크숍 참석 인원 최소 매출 손실 (트리 DP)

## 1. 문제 요약

- 트리 구조 조직도(루트 = 1번 CEO)가 주어짐.
- 각 "팀장 + 직속 팀원들"이 하나의 팀이며, **모든 팀은 최소 1명 이상 워크숍에 참석**해야 함.
- 참석 인원의 **하루평균 매출액 합을 최소화**하는 것이 목표.
- 정답은 2³¹-1 이하 보장, 노드 수는 최대 300,000.

핵심 구조: "부모 노드 + 그 직속 자식들 전체" = 하나의 그룹(팀)이고, 이 그룹마다 최소 1명 선택 제약이 걸리는 **트리 위의 최소 가중치 커버링 문제**.

---

## 2. 올바른 풀이 방향 (트리 DP, 2-state)

각 노드마다 두 가지 상태를 계산한다.

- `dp[node][1]` : node를 **선택**했을 때, node의 서브트리 전체(자기 자신 포함 모든 자손)를 커버하는 최소 비용
- `dp[node][0]` : node를 **선택 안** 했을 때, node의 서브트리 전체를 커버하는 최소 비용

### 점화식

```
dp[node][1] = sales[node] + Σ min(dp[child][0], dp[child][1])   (자식은 자유롭게 선택)

dp[node][0] = Σ min(dp[child][0], dp[child][1])
              + (자식 중 아무도 "선택 상태"를 고르지 않았다면 minExtra 추가)

minExtra = min( dp[child][1] - dp[child][0] )  (선택 쪽이 더 비싼 자식들 중 스왑 비용 최소값)

리프(자식 없음): dp[node][0] = 0  (팀 자체가 없으므로 제약 없음)
```

**왜 minExtra 보정이 필요한가?**
`dp[node][0]`일 때 각 자식이 개별적으로 더 싼 쪽(`min(dp0,dp1)`)을 고르면, 자식들이 전부 미선택(dp0)을 고르는 상황이 생길 수 있음 → "이 팀에서 아무도 안 뽑힘" 상태가 되어 제약 위반. 이를 막기 위해 **선택으로 바꾸는 데 추가비용이 가장 적은 자식 1명**을 강제로 선택 상태로 스왑.

### 최종 답

```
answer = min(dp[1][0], dp[1][1])   // 1번 = CEO = 루트
```

### `dp[cur][1]`의 의미 (헷갈렸던 부분)

`dp[cur][1]`은 **cur 한 명만의 비용이 아니라**, cur을 선택한 전제 하에 **cur을 뿌리로 하는 서브트리 전체(자손 모두 포함)를 커버하는 최소 비용의 총합**이다. 재귀적으로 자식의 dp 값이 이미 그 자식의 서브트리 정보를 다 포함하고 있으므로, 부모의 dp에는 그 아래 모든 노드의 최적화 결과가 누적되어 있다.

### 자식이 많을수록 sum이 커지는 이유 (헷갈렸던 부분)

"팀당 최소 1명" 조건은 트리 전체에서 **팀장+직속자식 단위마다 개별적으로** 걸리는 제약이다. 즉 "이 팀 하나만 1명 뽑으면 끝"이 아니라, 자식이 또 다른 팀의 팀장이면 그 자식의 서브트리도 별도로 커버해야 한다. 그래서 커버해야 할 서브트리(제약)가 늘어날수록 총 비용 합이 느는 것은 정상 동작이며 버그가 아니다. (단, 순수 팀원인 리프 자식은 `dp[leaf][0]=0`이라 sum에 기여하지 않는다.)

## 3. 시행착오 / 잘못된 접근들

### 3-1. 팀 개수 크기의 1차원 dp 배열로 접근 (❌)
`dp[group+1]`처럼 팀 개수 기준으로 배열을 잡는 방식은 이 문제 구조(노드별 선택/비선택 상태)를 표현할 수 없음. **노드마다 2개 상태**를 가져야 함.

### 3-2. 인접 2개(a,b)만 비교하는 페어와이즈 DP (❌)
"연결점 vs 이전 그룹 최소값" 식으로 자식을 2개씩만 비교하는 방식은, **자식이 3명 이상**인 노드에서 정보가 소실되어 틀림. 팀 전체(자식 몇 명이든)를 하나의 그룹 제약으로 다뤄야 함.
→ 다만 자식들을 **순차적으로 접어나가는(fold)** 형태(`already`/`notYet` 러닝 변수 유지)로 일반화하면 표준 2-state DP와 동치가 될 수 있음.

### 3-3. 루트 인덱스 오류
`tree[i+1]`로 노드를 채워놓고 `tree[0]`을 참조 → NPE. 루트는 항상 **1번 노드**(`tree[1]`, `dp[1]`)여야 함. `dp[0]`을 참조하면 계산된 적 없는 값(0)을 쓰게 되어 오답.

### 3-4. BFS로 순회 순서 만들 때 리프 스킵 (❌)
```java
if(visited[y] || g[y].size()==0) continue;  // 자식 없는 노드를 큐에서 제외
```
리프를 큐/`order`에서 빼면 그 노드의 dp가 계산되지 않아 기본값 0으로 남고, 부모가 이 값을 그대로 가져다 써서 **해당 직원의 매출액이 통째로 누락**됨. 방문 여부만 체크하고 자식 유무로 거르면 안 됨.

### 3-5. BFS 역순 루프에서 인덱스 혼동
```java
for (int cur : g[i]) { ... }        // ❌ i는 order의 인덱스일 뿐 노드 번호가 아님
dp[order[i]][1] = sales[order[i]-1] + sum;  // 저장은 order[i]로 하면서 순회는 g[i]로 함 (불일치)
```
→ `g[order[i]]`(또는 `node = order[i]`로 변수를 만들어 `g[node]`)로 통일해야 함. 이 실수는 `g[0]`이 `null`이라 `i=0` 시점에 NPE까지 유발할 수 있음.

### 3-6. int 오버플로우
`sales` 최대 10,000 × 노드 최대 300,000 → 중간 누적합이 int 범위(약 21억)를 초과할 수 있음. 최종 답은 `2^31-1` 이하로 보장되지만 **중간 dp 값**은 보장되지 않으므로 `dp` 배열과 연산은 `long`으로 처리. 최종 반환값만 필요 시 `(int)` 캐스팅.

### 3-7. 재귀 DFS의 스택 오버플로우 위험
노드 수가 최대 300,000이고 체인형(일렬) 트리가 가능 → 재귀 깊이 최대 30만 → `StackOverflowError` 위험. 해결책:
- BFS로 방문 순서(`order`)를 구한 뒤 **역순으로 반복 처리**하는 바텀업 방식 사용, 또는
- 재귀를 꼭 써야 한다면 스택 크기를 크게 잡은 별도 Thread에서 실행.

## 4. 개념 정리

### 바텀업 vs 탑다운
- 둘 다 **"자식의 dp 값이 먼저 확정되어야 부모의 dp를 계산할 수 있다"**는 의존성 자체는 동일.
- **바텀업**: BFS로 방문 순서를 구한 뒤 그 순서를 뒤집어(리프 쪽 → 루트 쪽) 반복문으로 처음부터 끝까지 전부 계산. 재귀 없음 → 스택 오버플로우 안전.
- **탑다운**: 루트에서 `getDp(1, 0/1)`을 "요청"하면 필요한 자식을 재귀로 그때그때 호출, 이미 계산된 `(node, state)`는 캐시(`computed`)에서 바로 반환. 이 문제는 트리 구조라 각 `(node,state)`가 정확히 1번씩만 필요하므로 캐시 유무와 무관하게 시간복잡도는 동일(O(N)). 다만 재귀 깊이 문제는 그대로 남음.
- `for(i=n-1; i>=0; i--)`로 도는 반복문은 "뿌리에서 시작"하는 게 아니라, **BFS 방문 순서(`order`)를 거꾸로 읽어 리프 쪽부터 처리**하는 바텀업 로직임.

### 시간복잡도
- DFS/트리 순회는 각 노드 1번, 각 간선 1번씩만 방문 → **O(V+E) = O(N)** (트리는 E = N-1).
- 공간복잡도도 dp 배열, 자식 리스트 모두 O(N). 재귀 버전은 스택 깊이 O(N) 추가.

## 5. 최종 정답 코드 (바텀업 버전)

```java
static long solution(int[] sales, int[][] links) {
    int n = sales.length;

    List<Integer>[] g = new ArrayList[n + 1];
    for (int i = 1; i <= n; i++) g[i] = new ArrayList<>();
    for (int[] link : links) g[link[0]].add(link[1]);

    // BFS로 "루트에서 가까운 순서" 기록
    int[] order = new int[n];
    boolean[] visited = new boolean[n + 1];
    Deque<Integer> q = new ArrayDeque<>();
    q.add(1);
    visited[1] = true;
    int idx = 0;
    while (!q.isEmpty()) {
        int x = q.poll();
        order[idx++] = x;
        for (int y : g[x]) {
            if (visited[y]) continue;   // 리프라도 스킵하면 안 됨
            visited[y] = true;
            q.add(y);
        }
    }

    long[][] dp = new long[n + 1][2];

    // order 역순 = 리프 쪽부터 루트 방향으로 처리 (바텀업)
    for (int i = n - 1; i >= 0; i--) {
        int node = order[i];
        long sum = 0;
        long minExtra = Long.MAX_VALUE;
        boolean anySelected = false;

        for (int child : g[node]) {
            long c0 = dp[child][0], c1 = dp[child][1];
            sum += Math.min(c0, c1);
            if (c1 <= c0) anySelected = true;
            else minExtra = Math.min(minExtra, c1 - c0);
        }

        dp[node][1] = sales[node - 1] + sum;

        if (g[node].isEmpty()) dp[node][0] = 0;
        else if (anySelected) dp[node][0] = sum;
        else dp[node][0] = sum + minExtra;
    }

    return Math.min(dp[1][0], dp[1][1]);
}
```

## 6. 체크리스트 (다음 트리 DP 문제 풀 때)

- [ ] 루트 번호가 1인지, dp 배열/순회 모두 그 번호로 일관되게 접근하는지
- [ ] 순회용 인덱스(`i`)와 실제 노드 번호(`order[i]`)를 혼동하지 않았는지
- [ ] 리프도 순회 대상에서 빠지지 않는지
- [ ] 값 범위(오버플로우 가능성) 확인 후 필요시 long 사용
- [ ] 재귀 깊이가 N까지 갈 수 있는 트리인지 확인 → 반복문 방식 고려
- [ ] "그룹 제약"이 있는 DP라면 그룹 내 원소 수와 무관하게(2개든 N개든) 일반화된 형태로 점화식을 세웠는지
