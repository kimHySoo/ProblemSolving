# 버전 관리 리스트 구현 회고

## 문제 개요

얕은 복사(shallow copy)와 깊은 복사(deep copy)를 지원하는 리스트 자료구조를 구현한다.
복사 이후 원본/복사본에 업데이트가 발생해도 각자의 독립성을 유지해야 한다.

---

## 내 아이디어

- 깊은 복사 시 새 노드를 생성하고 `par[]`로 부모를 가리킴
- 얕은 복사 시 동일한 노드를 그대로 참조
- 업데이트는 각 노드의 LinkedList에 `(index, value, version)` 형태로 기록
- `element()` 조회 시 `par[]` 체인을 타고 올라가며 버전 기준으로 필터링

아이디어 자체는 올바른 방향이었다.

---

## 구현에서 틀린 부분

### 버그 1: 체인 순회 방향

**문제**

조상→현재 순서로 `ans`를 갱신해야 하는데, 현재→조상 순서로 순회해서
조상의 오래된 값이 자식의 최신 값을 덮어쓰는 문제가 발생했다.

```java
// 기존: 현재 노드 먼저 순회
Node cur = ll[v].head.next;
while(cur != null) {
    if(cur.idx == mIndex) ans = cur.value;
    cur = cur.next;
}
// 이후 부모 체인 순회 → 부모의 값이 ans를 덮어씀
while(true) {
    v = par[v];
    ...
}
```

**수정**

`chain[]` 배열에 현재→루트 경로를 저장한 뒤, 루트→현재 역순으로 적용한다.
자식에 가까울수록 나중에 덮어써서 자식의 업데이트가 최종값이 된다.

---

### 버그 2: 버전 컷오프가 하나뿐

**문제**

다단계 깊은 복사에서 `version = arr[2]`(현재 노드의 생성 버전) 하나로
모든 조상 노드를 필터링했다.

```
A(ver=1) → deepCopy → B(ver=3) → deepCopy → C(ver=5)
```

```java
int version = arr[2]; // C의 버전 = 5

// A 업데이트 필터링할 때도 version=5 사용 → 틀림
// A는 B가 복사된 ver=3 기준으로 잘라야 함
while(cur != null && cur.version < version) { ... }
```

**수정**

`nodeVer[]` 배열을 추가해 각 노드가 생성된 버전을 저장한다.
부모 노드를 볼 때는 직계 자식이 복사된 시점(`nodeVer[자식]`)을 컷오프로 사용한다.

```
A 업데이트 컷오프 → nodeVer[B] = 3
B 업데이트 컷오프 → nodeVer[C] = 5
C 업데이트 컷오프 → 제한 없음
```

---

### 버그 3: cur 변수 재사용 문제

**문제**

부모 체인 순회 중 `cur`가 `null`이 된 상태에서 현재 노드 순회를 시도했다.

```java
// 부모 순회가 끝나면 cur = null
while(true) {
    v = par[v];
    cur = ll[v].head.next; // 마지막엔 null로 끝남
    ...
}
// 아래 while은 cur==null이라 실행되지 않음
while(cur != null) { // 현재 노드 순회가 실제로는 건너뜀
    if(cur.idx == mIndex) ans = cur.value;
}
```

**수정**

현재 노드 순회 전 `cur`를 명시적으로 재세팅한다.

```java
cur = ll[arr[1]].head.next; // 현재 노드로 다시 세팅
while(cur != null) {
    if(cur.idx == mIndex) ans = cur.value;
    cur = cur.next;
}
```

## 더 나은 구현과의 차이

| 항목 | 내 구현 | 개선된 구현 |
|---|---|---|
| 업데이트 저장 구조 | 전체를 하나의 LinkedList에 시간순 저장 | 인덱스별로 분리해서 List에 저장 |
| element() 조회 | 전체 업데이트 순회 O(N) | 해당 인덱스 로그만 꺼내 이진 탐색 O(log N) |
| 버전 필터링 | LinkedList 선형 탐색 | ArrayList + 이진 탐색 |

### 핵심 차이

내 구현은 `ll[nodeIdx]`에 모든 인덱스의 업데이트를 섞어서 저장하기 때문에,
조회 시 `mIndex`가 일치하는지 전체를 순회해야 한다.

개선된 구현은 `updatesByIndex`로 인덱스별로 분리 저장하기 때문에,
조회 시 해당 인덱스 로그만 O(1)으로 꺼내고 이진 탐색으로 O(log N) 필터링이 가능하다.

update 횟수가 많을수록 두 구현의 속도 차이는 기하급수적으로 커진다.

---

## 요약

아이디어(`par[]` 체인 + 버전 필터링)는 올바른 방향이었고 **구현 실수**가 원인이었다.

- 순회 방향 (현재→조상 X, 조상→현재 O)
- 단계별 버전 컷오프 (`nodeVer[]` 누락)
- `cur` 변수 재사용으로 인한 현재 노드 순회 누락

속도 측면에서는 업데이트 로그를 인덱스별로 분리 저장하고 이진 탐색을 적용하는 구조가 근본적으로 더 효율적이다.
