# 백준 32175 - 풀이 정리 및 실수 모음

## 문제 유형
- **DP (동적 프로그래밍)** - 경우의 수 세기
- **코인 교환 문제(Coin Change - Count)** 와 동일한 패턴

---

## ❌ 내가 틀렸던 이유들

### 1. 베이스 케이스 누락 (`dp[0] = 1`)
가장 핵심적인 실수. `dp[0] = 1`을 설정하지 않으면 모든 경우의 수가 0이 된다.

- `v == 0`은 컵을 딱 맞게 채운 경우 → **1가지**로 세야 함
- `dp[0] = -1`로 초기화된 상태에서 베이스 케이스가 없으면, for문을 돌며 `dfs(-cup[i])`를 호출해 전부 0을 반환받아 `dp[0] = 0`이 되어버림

```java
Arrays.fill(dp, -1);
dp[0] = 1; // ✅ 반드시 필요!
```

### 2. `visited` 배열과 `dp` 배열을 이중으로 관리
`visited` 배열을 따로 두면 로직이 복잡해지고 버그가 생기기 쉽다.
`dp[v] != -1`로 메모이제이션 체크를 한 번에 처리하면 된다.

```java
// ❌ 복잡한 방식
static boolean[] visited;
if(visited[v]) return dp[v];
visited[v] = true;

// ✅ 깔끔한 방식
if(dp[v] != -1) return dp[v];
```

### 3. `dp` 초기화 로직에서 인덱스와 컵 크기 혼동
```java
// ❌ 잘못된 코드 - 컵의 값(cup[i])이 아닌 배열 인덱스(i)로 dp 설정
for(int i=0; i<n; i++) {
    cup[i] = Integer.parseInt(st.nextToken());
    if(i > h) continue;
    dp[i]++;  // cup[i]의 값이 아닌 i번째 dp를 증가시키는 실수
}
```

### 4. `int` 오버플로우
MOD 연산 전 덧셈에서 오버플로우가 날 수 있다. `long`을 사용해야 안전하다.

```java
// ❌ 위험
static int[] dp;

// ✅ 안전
static long[] dp;
```


## ✅ 최종 정답 코드

### Top-Down (메모이제이션)
```java
import java.io.*;
import java.util.*;

public class Main {
    static int[] cup;
    static long[] dp;
    static int n;
    static final long MOD = 1_000_000_007;

    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
        StringTokenizer st = new StringTokenizer(br.readLine());
        n = Integer.parseInt(st.nextToken());
        int h = Integer.parseInt(st.nextToken());

        cup = new int[n];
        dp = new long[h + 1];
        Arrays.fill(dp, -1);
        dp[0] = 1; // ✅ 베이스 케이스

        st = new StringTokenizer(br.readLine());
        for (int i = 0; i < n; i++) cup[i] = Integer.parseInt(st.nextToken());

        System.out.println(dfs(h));
    }

    static long dfs(int v) {
        if (v < 0) return 0;
        if (dp[v] != -1) return dp[v]; // ✅ 메모이제이션

        dp[v] = 0;
        for (int i = 0; i < n; i++) {
            dp[v] = (dp[v] + dfs(v - cup[i])) % MOD;
        }
        return dp[v];
    }
}
```

### Bottom-Up (권장)
```java
import java.io.*;
import java.util.*;

public class Main {
    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
        StringTokenizer st = new StringTokenizer(br.readLine());
        int n = Integer.parseInt(st.nextToken());
        int h = Integer.parseInt(st.nextToken());

        int[] cup = new int[n];
        st = new StringTokenizer(br.readLine());
        for (int i = 0; i < n; i++) cup[i] = Integer.parseInt(st.nextToken());

        long[] dp = new long[h + 1];
        final long MOD = 1_000_000_007;
        dp[0] = 1; // ✅ 베이스 케이스

        for (int j = 1; j <= h; j++) {
            for (int i = 0; i < n; i++) {
                if (j >= cup[i]) {
                    dp[j] = (dp[j] + dp[j - cup[i]]) % MOD;
                }
            }
        }

        System.out.println(dp[h]);
    }
}
```

### Bottom-Up 점화식 이해

**`dp[j]` = 높이 j를 정확히 채우는 경우의 수**

```
dp[0] = 1          (아무것도 안 쌓는 경우, 베이스 케이스)
dp[1] = ?
dp[2] = ?
...
dp[h] = 구하는 답
```

예시: 컵 크기가 [1, 2, 3]이고 h = 4라면

| j | 계산 | dp[j] |
|---|------|-------|
| 0 | 베이스 케이스 | 1 |
| 1 | dp[1-1]=dp[0]=1 | 1 |
| 2 | dp[2-1]+dp[2-2]=dp[1]+dp[0]=1+1 | 2 |
| 3 | dp[3-1]+dp[3-2]+dp[3-3]=2+1+1 | 4 |
| 4 | dp[4-1]+dp[4-2]+dp[4-3]=4+2+1 | 7 |

**점화식:**
```
dp[j] = Σ dp[j - cup[i]]   (j >= cup[i]인 모든 i에 대해)
```

---

### ⚠️ 범위 설정이 왜 그렇게 되어야 하는가

#### 바깥 루프: `j = 1` ~ `j <= h`

- `j = 0`은 이미 베이스 케이스로 `dp[0] = 1` 설정 완료, 건드리면 안 됨
- `j = h`까지 포함해야 최종 답 `dp[h]`가 계산됨 → `j < h`로 쓰면 답이 0

```java
for (int j = 1; j <= h; j++) { ... }  // ✅ h 포함
for (int j = 1; j < h; j++)  { ... }  // ❌ dp[h]가 계산 안 됨
```

#### 안쪽 조건: `if (j >= cup[i])`

- `dp[j - cup[i]`를 참조하므로 인덱스가 0 이상이어야 함
- `j < cup[i]`면 `j - cup[i] < 0` → 배열 범위 초과(ArrayIndexOutOfBoundsException)

```java
if (j >= cup[i]) {
    dp[j] += dp[j - cup[i]];  // j - cup[i] >= 0 보장됨
}
```

Top-Down에서 `if (v < 0) return 0;`으로 처리하던 것과 동일한 역할이다.

#### 배열 크기: `new long[h + 1]`

- 인덱스 0 ~ h까지 사용하므로 크기는 반드시 `h + 1`
- `new long[h]`로 선언하면 `dp[h]` 접근 시 ArrayIndexOutOfBoundsException

```java
long[] dp = new long[h + 1];  // ✅ 인덱스 0 ~ h 모두 사용 가능
long[] dp = new long[h];      // ❌ 인덱스 h 접근 불가
```

## 📚 알면 좋은 것들

### Top-Down vs Bottom-Up 비교

| | Top-Down (DFS + 메모이제이션) | Bottom-Up (반복문) |
|---|---|---|
| 코드 복잡도 | 높음 (재귀 구조) | 낮음 (이중 for문) |
| 스택 오버플로우 위험 | h가 크면 위험 | 없음 |
| 직관성 | 점화식 그대로 표현 가능 | 순서 고려 필요 |
| 속도 | 유사 | 캐시 친화적으로 약간 유리 |
| 추천 상황 | 일부 상태만 필요할 때 | 전체 상태가 필요할 때 |

### 메모이제이션 체크 관용 패턴
`visited` 배열 없이 dp 배열 하나로 처리하는 것이 표준이다.

```java
// dp 배열을 -1로 초기화하고 방문 여부로 활용
Arrays.fill(dp, -1);

static long dfs(int v) {
    if (dp[v] != -1) return dp[v]; // 이미 계산됨
    dp[v] = 0;
    // ... 계산 ...
    return dp[v];
}
```

### 코인 교환 패턴 (자주 나오는 유형)
이 문제는 **코인 교환 - 경우의 수** 유형과 완전히 동일하다.

- `dp[0] = 1` (아무것도 선택 안 하는 경우)
- `dp[j] += dp[j - coin[i]]` (j를 만드는 경우의 수)

이 패턴을 기억하면 비슷한 문제에 바로 적용 가능하다.

### MOD 연산 시 주의사항
- 덧셈 누적 시 `int` 범위를 넘을 수 있으므로 `long` 사용
- MOD를 취하는 시점: 덧셈 직후 or 반환 직전 (일관성 있게)

```java
dp[v] = (dp[v] + dfs(v - cup[i])) % MOD; // ✅ 덧셈 직후 MOD
```
