# BOJ 17136 색종이 붙이기 오답노트 / README

## 1. 문제 핵심

10x10 종이 위의 `1`들을 모두 덮어야 한다. 사용할 수 있는 색종이는 `1x1, 2x2, 3x3, 4x4, 5x5`가 각각 5장씩이며, 색종이는 **정사각형 모양만 가능**하다. 목표는 모든 `1`을 덮는 데 사용하는 색종이 수의 최솟값을 구하는 것이다. 불가능하면 `-1`을 출력한다.

이 문제의 본질은 단순 구현이 아니라, **백트래킹에서 상태를 정확히 정의하고 중복 탐색을 막는 것**이다.

---

## 2. 내가 이 문제에서 잘못 생각한 점

### 2-1. 한 DFS 호출에서 여러 기준 칸을 동시에 처리하려고 한 점
처음 코드에서 가장 큰 문제는, 아직 덮이지 않은 `1`을 발견한 뒤에도 계속 전체 맵을 순회했다는 점이다.

백트래킹은 현재 상태에서

- 아직 처리되지 않은 첫 번째 칸 하나를 고르고
- 그 칸에 대해 가능한 선택지를 모두 시도한 뒤
- 재귀로 내려가야 한다.

그런데 한 DFS 호출에서 여러 칸을 계속 돌면,

- 현재 상태에서의 선택 기준이 여러 개가 되고
- 같은 상태를 다른 순서로 중복 탐색하게 되며
- 탐색 트리 구조가 무너진다.

즉, 이 문제는 **첫 번째 미처리 `1` 칸 하나만 잡고 분기한 뒤 바로 return** 해야 한다.

---

### 2-2. 색종이를 붙였는지와 원래 종이 상태를 분리해서 생각하지 못한 점
처음에는 `map[r][c] == 1`만 확인하면 그 칸을 덮을 수 있다고 생각하기 쉽다. 하지만 실제로는

- 원래 종이 상태는 `map`
- 현재 DFS에서 이미 덮은 상태는 `visited`

로 구분해야 한다.

즉, 어떤 정사각형을 붙일 수 있는지 검사할 때는

- 범위를 벗어나지 않는지
- 모두 `1`인지
- 이미 다른 색종이로 덮인 칸이 아닌지

를 모두 확인해야 한다.

`visited`를 보지 않으면 이미 덮은 칸 위에 또 덮는 잘못된 배치가 가능해진다.

---

### 2-3. “넓이만 맞으면 될 것”처럼 생각한 점
이 문제는 넓이 합만 맞는다고 해결되지 않는다.

예를 들어 어떤 모양은 넓이상으로는 덮을 수 있어 보여도,

- 색종이는 직사각형이 아니라 정사각형뿐이고
- 이어진 모양이 아니면 큰 색종이를 못 쓰고
- 작은 색종이 개수 제한 때문에 불가능할 수 있다.

즉, **넓이 계산은 보조적인 감각일 뿐이고, 실제 배치는 모양 단위로 판단해야 한다.**

---

### 2-4. 가지치기의 기준을 약하게 잡은 점
색종이를 이미 최적해보다 많이 썼다면 더 내려갈 필요가 없다.

그런데 가지치기를 애매하게 잡으면 필요 없는 분기를 많이 탐색하게 된다.

예를 들어 현재까지 사용한 색종이 수가 `used`이고, 현재 최적값이 `ans`라면

```java
if (used >= ans) return;
```

처럼 바로 자르는 게 맞다.

동일한 개수여도 더 좋은 답이 나올 수 없으므로 버려야 한다.

---

## 3. 이 문제에서 특히 주의해야 할 부분

### 3-1. 기준 칸은 항상 “첫 번째 미처리 1”이어야 한다
이렇게 해야 중복 탐색이 줄어든다.

정석 구조:

1. 위에서부터, 왼쪽에서부터 첫 번째 `1`을 찾는다.
2. 그 칸에 놓을 수 있는 색종이를 시도한다.
3. 재귀 호출 후 복구한다.
4. 그 칸에 대한 분기가 끝났으면 바로 return 한다.
5. 끝까지 `1`을 못 찾았으면 모든 칸을 덮은 것이므로 답 갱신.

---

### 3-2. 가능 검사(`chk`)와 상태 변경(`cover`)는 정확히 분리해야 한다
가능 검사 함수는 “놓을 수 있나?”만 판단해야 하고, 실제 상태 변경은 별도 함수에서 해야 한다.

이 둘이 섞이면 복구가 꼬이기 쉽다.

---

### 3-3. 복구(backtracking restore)는 100% 대칭이어야 한다
예를 들어

```java
cover(..., true);
paper[k]--;
dfs(...);
paper[k]++;
cover(..., false);
```

처럼 **내려가기 전에 바꾼 상태를 정확히 되돌려야 한다.**

하나라도 빠지면 이후 모든 탐색이 오염된다.

---

### 3-4. 큰 색종이부터 시도하는 것이 유리하다
정답성에 필수는 아니지만, 일반적으로 큰 색종이부터 놓으면

- 적은 장수로 빨리 덮을 가능성이 높고
- 좋은 답을 빨리 찾게 되어
- 가지치기가 더 강해진다.

그래서 보통 `5 -> 1` 순서로 시도한다.

---

## 4. 현재 코드 평가

### 장점
- DFS + 백트래킹 구조 자체는 잘 잡았다.
- 색종이 개수를 `int[] x`로 관리한 것은 적절하다.
- `cover`와 `chk`를 함수로 분리하려고 한 방향은 좋다.
- 색종이 장수 최솟값을 갱신하는 문제라는 점을 잘 파악했다.

### 아쉬운 점
1. **첫 번째 미처리 칸만 처리하고 return 해야 하는데, 그 점을 놓치기 쉬운 구조였다.**
2. `chk()`에서 `visited`를 보지 않으면 겹쳐 붙이는 잘못된 배치가 가능하다.
3. 가지치기 조건이 조금 약하다.
4. 탐색 순서를 `1 -> 5`로 두면 경우에 따라 비효율적이다.
5. `visited` 방식은 가능하지만, 이 문제는 `map` 자체를 0/1로 바꾸는 방식이 더 직관적일 때가 많다.

전체적으로 보면, **풀이 방향은 맞았고 백트래킹 상태 정의에서 한두 군데 중요한 실수를 한 케이스**다. 즉, 접근 자체가 틀린 건 아니고, 백트래킹의 정석 패턴을 조금 더 몸에 익히면 충분히 해결 가능한 수준이다.

---

## 5. 정석적인 풀이 아이디어

이 문제의 정석은 다음과 같다.

### 상태
- 현재 종이 상태 (`map`)
- 각 크기 색종이의 남은 개수 (`paper[1..5]`)
- 현재까지 사용한 색종이 수 (`used`)
- 현재 찾은 최소값 (`ans`)

### 재귀 로직
1. `used >= ans`면 가지치기.
2. 아직 덮지 않은 첫 번째 `1`을 찾는다.
3. 못 찾으면 전부 덮은 것이므로 `ans = min(ans, used)`.
4. 찾았으면 그 칸에서 `5x5`부터 `1x1`까지 가능한 색종이를 시도한다.
5. 붙이고 → 재귀 → 복구.
6. 그 칸에 대해 모든 분기를 시도했으면 return.

이 구조가 가장 깔끔하다.

---

## 6. 정석 코드

아래 코드는 `visited` 없이 `map` 자체를 바꾸는 방식의 정석 풀이이다.

```java
import java.io.*;
import java.util.*;

public class Main {
    static int[][] map = new int[10][10];
    static int[] paper = {0, 5, 5, 5, 5, 5};
    static int ans = Integer.MAX_VALUE;

    public static void main(String[] args) throws Exception {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
        for (int i = 0; i < 10; i++) {
            StringTokenizer st = new StringTokenizer(br.readLine());
            for (int j = 0; j < 10; j++) {
                map[i][j] = Integer.parseInt(st.nextToken());
            }
        }

        dfs(0);
        System.out.println(ans == Integer.MAX_VALUE ? -1 : ans);
    }

    static void dfs(int used) {
        if (used >= ans) return;

        for (int i = 0; i < 10; i++) {
            for (int j = 0; j < 10; j++) {
                if (map[i][j] == 1) {
                    for (int size = 5; size >= 1; size--) {
                        if (paper[size] == 0) continue;
                        if (!canAttach(i, j, size)) continue;

                        attach(i, j, size, 0);
                        paper[size]--;
                        dfs(used + 1);
                        paper[size]++;
                        attach(i, j, size, 1);
                    }
                    return;
                }
            }
        }

        ans = Math.min(ans, used);
    }

    static boolean canAttach(int r, int c, int size) {
        if (r + size > 10 || c + size > 10) return false;

        for (int i = r; i < r + size; i++) {
            for (int j = c; j < c + size; j++) {
                if (map[i][j] != 1) return false;
            }
        }
        return true;
    }

    static void attach(int r, int c, int size, int value) {
        for (int i = r; i < r + size; i++) {
            for (int j = c; j < c + size; j++) {
                map[i][j] = value;
            }
        }
    }
}
```

## 7. 왜 `visited`보다 `map` 수정 방식이 더 많이 쓰이나?

네 방식처럼 `visited`를 따로 둬도 풀 수 있다. 하지만 이 문제에서는 `map`을 직접 바꾸는 방식이 더 직관적인 경우가 많다.

이유는 다음과 같다.

### 7-1. 상태가 하나로 합쳐진다
`map[i][j] == 1`이면 아직 덮어야 하는 칸, `0`이면 이미 덮였거나 원래 빈 칸이다.

즉, 현재 상태를 `map` 하나로 표현할 수 있어서 관리가 쉽다.

### 7-2. 가능 검사에서 실수할 여지가 줄어든다
`visited`를 따로 두면

- `map == 1`
- `visited == false`

를 둘 다 봐야 한다.

하지만 `map` 수정 방식이면 `map == 1`만 보면 된다.

### 7-3. 복구가 더 직관적이다
붙일 때 0으로 바꾸고, 복구할 때 1로 바꾸면 된다.

이 문제처럼 “덮인 칸”과 “안 덮인 칸”이 명확한 문제는 `map` 직접 수정이 실수 방지에 좋다.

## 8. 이 문제를 통해 꼭 익혀야 할 백트래킹 패턴

### 패턴 1. 첫 번째 미해결 지점 하나만 잡고 분기
이건 스도쿠, 색종이, N-Queen, 퍼즐류 문제에서 매우 자주 나온다.

### 패턴 2. 상태 변경 / 재귀 / 복구의 대칭 구조
```java
상태 변경
재귀 호출
상태 복구
```
이 구조가 무너지면 백트래킹은 거의 항상 틀린다.

### 패턴 3. 최적화 문제에서는 pruning이 사실상 필수
최솟값/최댓값 문제는 현재 상태가 이미 최적해보다 나쁘면 즉시 잘라야 한다.

### 패턴 4. 가능성 검사 함수는 정확해야 한다
특히 범위, 중복, 이미 사용된 상태를 빠뜨리지 않아야 한다.

---

## 9. 한 줄 정리

이 문제에서 가장 중요했던 실수는 다음 두 가지다.

1. **한 DFS 호출에서 첫 번째 미처리 칸 하나만 처리해야 하는데, 여러 칸을 계속 탐색하려고 한 점**
2. **색종이를 붙일 수 있는지 검사할 때 이미 덮인 상태를 정확히 반영하지 못한 점**

이 두 가지만 정확히 잡으면, 이 문제는 전형적인 백트래킹 정석 문제로 정리된다.

---

## 10. 앞으로 비슷한 문제를 만났을 때 체크리스트

- 현재 상태를 무엇으로 표현할 것인가?
- 미해결 지점을 하나만 골라 분기하고 있는가?
- 가능 검사 함수가 범위/중복/조건을 모두 확인하는가?
- 상태 복구가 완전히 대칭적인가?
- 현재 최적해보다 나빠졌을 때 바로 가지치기 하는가?
- 큰 선택지부터 시도해 pruning을 강화할 수 있는가?

이 체크리스트를 습관처럼 적용하면 백트래킹 정확도가 훨씬 올라간다.
